Indexing - performance decisions and tradeoffs

Partial indexes over full indexes

The core decision: only index rows that are actively being worked on. idx_stages_polling covers PENDING and PROCESSING rows only. idx_stages_processing_expires_at covers PROCESSING rows with a non-null expires_at only.

The tradeoff is deliberate - historical COMPLETED rows, which will be the vast majority at scale, are excluded entirely. A standard index on status would grow with every verification ever processed, eventually becoming slow and memory-hungry. The performance gain - partial indexes stay small and fit in RAM regardless of how large the table grows. Worker query time stays flat instead of degrading as data accumulates.

Ordered index on outbox_events

Performance gain idx_outbox_polling on created_at ASC lets the relay grab the 50 oldest rows in arrival order without a sort operation. Tradeoff - insert work pays a small overhead to maintain the index (negligible here because the table stays near-empty via deletion).

SKIP LOCKED only scales because the indexes are shaped correctly

SKIP LOCKED tells Postgres to skip locked rows and return the next available ones. Performance gain - with the right index, Postgres finds the next available rows quickly; many worker pods grab disjoint batches without blocking each other. Tradeoff if mis-shaped - without the aligned index, this degrades to row-by-row scan under load.

Core tradeoff in one sentence. Slightly more complex index maintenance and a few extra bytes per write - in exchange, the performance gain: flat query time at any table size and contention-free concurrency across worker pods.

UUIDv4 - performance and security tradeoffs

What we gained

Performance gain - UUIDs are generated in memory with Uuid::new_v4(); no DB round-trip. Multi-stage transactions do not stall waiting for an ID. Sequential IDs under concurrency trade off serial contention - every insert waits on a locked sequence.

Performance gain - global uniqueness without coordination across API/worker pods or shards.

Security gain - 122 bits of entropy makes the session ID act as a bearer token; sequential IDs make enumeration trivial.

What we gave up

Tradeoff - 16 bytes per key vs 8, and more B-tree fragmentation than sequential keys because inserts land at random positions (more page splits, higher index maintenance).

Tradeoff in plain terms. Extra disk and mild fragmentation - performance and ops gains: no sequence bottleneck, no cross-pod ID coordination, IDOR resistance, safer sharding. For this workload, the tradeoff skews heavily toward UUIDs.

Transaction flows - performance decisions and tradeoffs

What SKIP LOCKED achieves in the outbox relay

Firstly, without SKIP LOCKED, parallel workers become a single-file line at the database.

Multiple workers polling the same table would block each other. Worker A locks a row; Worker B tries to read the same row and waits - held at the database level until Worker A releases the lock. Under high concurrency that turns into a queue of workers stacking up behind each other, serializing what should be parallel work.

SKIP LOCKED flips that behavior. Instead of waiting, a worker skips rows that are already locked and moves to the next available one. Two workers polling at once get different rows with no coordination. Ten workers get ten different batches. A hundred pods never touch the same row. The table acts as a self-distributing work queue - Postgres partitions work through row-level locking alone.

That is why both the Outbox Relay and the Lease Recovery Worker scale horizontally. Add pods and throughput tracks linearly: each pod works on rows no other pod is touching.

The tradeoffs

No ordering guarantee. SKIP LOCKED does not promise insertion order. If Worker A holds row 1, Worker B skips it and takes row 2 - row 2 can finish first. For the outbox that is usually fine: the ordered index on created_at biases work toward oldest-first, but strict FIFO is not enforced.

At-least-once, not exactly-once. If a worker reads a row, publishes it, then crashes before deleting it, the next worker will pick the row up again. The design assumes that - idempotency keys on the consumer absorb duplicates. Duplicates still happen; the system tolerates them, it does not erase them.

Locks for the whole transaction. While a worker is publishing to RabbitMQ it still holds the Postgres row lock. A slow or hung publish keeps those rows unavailable to every other worker until the transaction ends. That is why the 50ms publish timeout exists - it caps how long a stalled connection can hoard locks.

Invisible rows under heavy contention. If every row in a candidate batch is already locked, SKIP LOCKED returns an empty set instead of waiting. A worker may poll, find nothing, and sleep even though work still sits in the table - momentarily claimed by others. Expected behavior, not a bug; under extreme contention throughput is not perfectly linear.

Session creation - one transaction, bulk insert

The session row and all five stage rows are written in one transaction using unnest (one SQL statement vs five round-trips). Performance gain - one network round-trip to Postgres regardless of stage count. Tradeoff - slightly larger write transaction; lock time stays small, and latency drops materially.

Stage submission - conditional outbox inside one transaction

The LOCKED → PENDING update, dependency check, and outbox insert share one transaction scope - no second connection, no race between read and insert. Tradeoff - transaction stays open slightly longer for the conditional read. Performance gain - removes dual-write bugs where the stage updates but the outbox row never lands.

Outbox relay - row locks held only as long as publish takes

The relay locks up to 50 rows, publishes, deletes only ACKed rows, commits; per-row publish timeout caps lock duration. Tradeoff - slow RabbitMQ extends lock hold time up to the timeout cap. Performance gain - timeouts prevent unbounded lock hoarding; timed-out rows retry next cycle without blocking the batch indefinitely.

Stage claim - atomic PENDING → PROCESSING with lease

Single update claims the stage and sets expires_at; zero rows means another worker won. Tradeoff - one round-trip per message. Performance gain - no check-then-act gap where two workers could claim the same stage.

Stage completion and successor unlock - same transaction

Completing a stage and enqueueing the successor outbox row are atomic. Tradeoff - slightly longer transaction. Performance gain - sequencing enforced in the database without orchestrator, scheduler, or inter-stage delay; rollback never queues a successor by mistake.

Session sweep - batch update, single round-trip

Session sweep uses = ANY($1) over a batch of session IDs. Performance gain - O(1) statements instead of N round-trips for N sessions; sweep cost stays flat with batch size. Tradeoff - three conditional updates per call (mutually exclusive targets, so no double-writes).

Lease recovery - CTE-based single-statement update

Expired rows move to PENDING or FAILED in one CTE; lock acquisition and write share one execution. Tradeoff - more complex SQL to maintain. Performance gain - no window where two recovery workers claim the same expired row.

Timeout decisions - performance impact and tradeoffs

Publish timeout (50ms, outbox relay)

While publishing, the relay holds row locks. Performance gain - 50ms cap releases locks if the broker hangs, so Postgres stays usable. Tradeoff - a slow-but-healthy broker may be skipped and retried on the next poll; DB availability is prioritized over single-attempt publish speed.

Pool acquire timeout (10 seconds, API)

Performance gain - requests stop waiting before they pile up in RAM and OOM the pod. Tradeoff - some callers see fast 500s instead of eventually succeeding.

Worker database transaction timeout (10 seconds)

Long transactions hold row locks. Performance gain - hard ceiling forces nack/requeue and releases the connection. Tradeoff - transient DB slowness may cause extra requeues vs cascading lock contention.

Shutdown grace period (30 seconds, stage processor)

Tradeoff shaping - too short abandons in-flight work; too long slows deploys. Chosen gain - 30s balances clean shutdown against rollout speed.

Core tradeoff across timeouts. You pay occasional unnecessary retries. Performance gain - predictable behavior under degraded infra: fewer hung connections, less memory exhaustion, less lock pile-up.

Failure handling - performance decisions and tradeoffs

  • Retry limits with a hard ceiling - tradeoff: some jobs that might eventually succeed are abandoned. Performance gain: no infinite retry loops eating workers and DB connections.
  • 30-second delay queue - tradeoff: slower end-to-end retry. Performance gain: a bad job cannot tight-loop through the worker at full speed.
  • Reject to dead-letter - tradeoff: requires human replay from the DLQ. Performance gain: main queue stays healthy; poison messages leave the hot path.
  • Lease recovery in small batches - tradeoff: slower full recovery after mass failure. Performance gain: Postgres stays responsive while recovery ticks.
  • Poison pill permanent failure - tradeoff: gives up on toxic payloads. Performance gain: stops an infinite crash-resurrect loop across the pipeline.

Retry limits - performance decisions and tradeoffs

Stage processing retries (limit: 3)

Tradeoff - more retries help transient third-party blips; fewer retries evict toxic work sooner. At 3, a failed job spends ~90s in retry holds before quarantine. Performance gain - beyond that point, expected value of another attempt drops vs resource cost.

The 30-second retry delay

Performance gain - backpressure: sustained downstream failure cannot flood the main queue with the same failing job. Tradeoff - legitimate recovery waits longer between attempts.

Lease recovery retries (limit: 3)

Tradeoff - higher limits chase flaky infra longer; lower limits quarantine poison payloads faster. At 3, worst-case damage is bounded by ~3 minutes of recovery cadence before permanent failure. Performance gain - caps wasted CPU, DB, and queue time on jobs that never stabilize.

API execution

The API layer - what it does and how it flows

The API is a gateway.

It validates, locks data into Postgres, and hands off to workers.

It does no KYC work itself.

1. Every request passes through security before routing

The x-api-key header is checked using constant-time comparison - no timing attacks.

If the request carries an Idempotency-Key, Redis is checked first.

A duplicate request never reaches Postgres - the cached response is returned immediately.

Rate limiting is enforced in memory before any database or business logic is touched.

2. All incoming JSON goes through a two-pass validation

Serde validates structure first - wrong types are rejected immediately.

The validator crate then enforces field rules: string lengths, enum bounds, required formats.

Phone numbers go through libphonenumber for E.164 verification.

Any failure returns a 422 with exact field-level errors.

The handler function never runs on invalid input.

3. Stage submission opens one transaction and does exactly three things

First, it updates the stage from LOCKED to PENDING.

If that update affects 0 rows - because the stage is already in progress or completed - the API returns 409 Conflict and stops.

No duplicate work enters the system.

Second, it evaluates the dependency rule.

Independent stages (Phone, Liveness, Face-Match) get an outbox_events row inserted immediately.

Dependent stages (Personal Info, Document) only get the outbox row if their predecessor is already COMPLETED - otherwise the insert is skipped and the worker handles the trigger later.

Third, the transaction commits and the API returns 202 Accepted.

The work is queued; the background engine takes over from here.

4. The pool has a hard acquire timeout

If the database is under load, the API waits a maximum of 10 seconds for a connection.

After that it returns a 500 and moves on.

Without this, requests pile up in memory until the pod crashes.

5. Status polling is a single endpoint

One GET /status call returns the parent session and all five stage rows in one indexed query.

The response maps input_data and response_data together into an ordered timeline - the frontend sees the full verification journey in one shot.

6. Errors always return the same shell

Every 4xx and 5xx includes a trace_id tied to that specific request's tracing span.

A client reports a bug, hands over the trace_id, and the exact database queries and validation errors for that millisecond are immediately retrievable from logs.

API documentation

API documentation is generated at compile time from the Rust structs and route macros directly.

The Swagger UI at /swagger-ui always reflects the actual code.

Global headers (every request)

  • x-api-key - required on all endpoints. Missing or invalid returns 401 immediately.
  • Idempotency-Key - optional, used on session start. If the key exists in Redis, the cached 201 response is returned without touching Postgres.

POST /v1/verify/session/start

No payload.

Creates one session row and five LOCKED stage rows in a single transaction.

Returns 201 with the new session_id, status ACTIVE, and a message.

Nothing is processed here - this just provisions the state machine.

Stage submissions - shared contract

All five share the same behavior: validate the payload, lock the stage to PENDING, insert an outbox event, return 202 Accepted.

The 202 is intentional - the work is queued, not complete.

  • POST .../stages/phone - { phone_number }. Validated against E.164 format via libphonenumber.
  • POST .../stages/personal-info - { first_name, last_name, dob, address }. dob is parsed as YYYY-MM-DD. String lengths are bounded.
  • POST .../stages/document - { document_type, document_image_b64, document_number }. document_type must be exactly passport, national_id, or drivers_license. Anything else is a 400.
  • POST .../stages/liveness - { liveness_image_b64 }.
  • POST .../stages/face-match - { face_image_b64 }.

Shared failure responses: 409 Conflict - stage is not LOCKED.

422 Unprocessable Entity - payload is structurally valid but fails business validation.

GET /v1/verify/{session_id}/status

No payload.

Returns the full verification journey in one response - session status (ACTIVE, COMPLETED, FAILED) plus all five stages in order.

Each stage includes its current status, the original input_data, the worker's response_data on success, and error_message on failure.

Error shell - always this shape

{
  "error": "machine_readable_code",
  "message": "Human readable hint",
  "details": { "field": "reason" },
  "trace_id": "016x_hex_string"
}

The trace_id maps directly to the tracing span for that request.

A reported 500 becomes a log lookup, not a debugging session.

Outbox relay worker

Firstly, why do we have an Outbox Relay architecture?

The API deliberately never talks to RabbitMQ directly.

When a stage is submitted, the API opens one Postgres transaction, writes the session data, and exits. If the API also tried to publish to RabbitMQ inside that same flow, you have a dual-write problem - the database write and the RabbitMQ publish are two separate operations with no shared atomicity. The database could commit but RabbitMQ could be down. Or RabbitMQ could receive the message but the database transaction rolls back. Either way, the two systems fall out of sync and a job is either lost or processed against data that was never actually saved.

The outbox table solves this by making the RabbitMQ intent part of the database transaction itself. The API writes the message payload into outbox_events in the exact same commit that writes the stage row. If the transaction commits, the event is guaranteed to exist. If it rolls back, the event disappears with it. At that point, RabbitMQ being up or down is completely irrelevant to the API.

The Outbox Relay exists as a separate worker because someone has to bridge that gap - read the committed rows from Postgres and actually deliver them to RabbitMQ. That’s its entire job. Nothing more.

What it does and how it flows

Single mission: move rows from outbox_events into RabbitMQ without losing a single one.

Isolated connection pool

The relay uses its own dedicated Postgres pool, separate from the main application pool.

A RabbitMQ outage that causes connections to hang can only exhaust the relay's pool - the API and stage processors keep running on the main pool, unaffected.

Polling with SELECT FOR UPDATE SKIP LOCKED

The worker polls outbox_events in batches of 50 (configurable).

SKIP LOCKED means multiple relay pods can poll the same table simultaneously without colliding - any row locked by one pod is instantly skipped by the others.

Pods never wait on each other.

Publish with a hard 50ms timeout

While publishing, the worker is holding row locks in Postgres.

If the RabbitMQ connection hangs, waiting indefinitely would freeze those locks.

Every publish attempt has a strict 50ms timeout - if it expires, the row is skipped and the lock is released immediately.

Database health is prioritized over message dispatch speed.

The batch cycle is one atomic transaction

  1. Fetch and lock up to 50 rows (configurable).
  2. Publish each to RabbitMQ.
  3. Collect the IDs of successfully acknowledged messages.
  4. Bulk DELETE only those successful rows.
  5. Commit.

If the pod loses power after RabbitMQ acknowledges but before the DELETE commits, the transaction rolls back, the locks drop, and another pod picks up the same rows.

At-least-once delivery, zero data loss.

Corrupt rows are skipped, not fatal

A malformed row causes the relay to log the error, skip that row, and continue publishing the rest of the batch.

One bad row never blocks the queue.

Smart polling cadence

If the outbox is empty, the worker sleeps for a full idle interval before polling again - no pointless database hammering.

On startup, a random jitter (0-5 seconds) staggers the first poll across pods, preventing a thundering herd against Postgres when a deployment rolls out.

Lease recovery worker

Firstly, why do we need a Lease Recovery Worker?

The Stage Processor claims a job by writing PROCESSING to the database, then does the work. That claim has no automatic expiry - once it’s written, Postgres holds that status until something explicitly changes it.

The problem is pods die. A Kubernetes node loses power, a container runs out of memory, a network partition isolates the pod - the Stage Processor is gone, but the database row still says PROCESSING. The RabbitMQ message was already ACKed when the job was claimed, so the broker has no copy to retry. The job is permanently stuck. No worker will ever pick it up again because nothing is polling for PROCESSING rows - only PENDING ones enter the queue.

The expires_at column turns the claim into a timed lease. When a worker claims a job, it writes a deadline. If the worker finishes, it updates the stage and the lease becomes irrelevant. If the worker dies, the lease expires and the row sits there with status = PROCESSING and expires_at in the past - a detectable, recoverable state.

The Lease Recovery Worker exists as the only process that watches for that exact condition. It scans for expired leases, resets the stage back to PENDING, and inserts a fresh outbox event so the job re-enters the pipeline. Without it, every crashed pod means a permanently lost job. With it, the system heals itself automatically on the next recovery tick.

What it does and how it flows

Single mission: find jobs that were claimed but never finished, and either resurrect them or permanently bury them.

Search and rescue - every 60 seconds (configurable)

The worker polls for stages where status = PROCESSING and expires_at is in the past.

A worker that claimed a job sets a 120-second lease - if that time has passed, the pod is definitively dead.

Rows are fetched in batches of 25 using SELECT FOR UPDATE SKIP LOCKED.

Small batches mean a mass node failure dropping thousands of jobs doesn't lock the entire table at once - recovery happens in surgical chunks.

Reclaim - reset and re-queue

For each expired row, the worker resets the status back to PENDING, increments lease_recovery_count, and inserts a new row into outbox_events - all in the same transaction.

The Outbox Relay picks up that event and pushes the job back into RabbitMQ for a Stage Processor to retry.

The system heals itself with no human intervention.

Poison pill quarantine - stop at the limit

Before resurrecting a job, the worker checks lease_recovery_count.

If it has hit the configured limit (3), the job is not retried - the status is permanently set to FAILED and no outbox event is inserted.

A payload that crashes the worker every single time is quarantined in the database.

The rest of the infrastructure is protected.

Transactional integrity and session sweep

The entire batch - resets, outbox inserts, and failure writes - happens inside one Postgres transaction.

If the database crashes mid-recovery, the transaction rolls back and the expired rows sit untouched until the next 60-second tick.

Before the transaction commits, the worker runs a session sweep - if any stage in a session was just permanently failed, the parent session status is updated to FAILED in the same commit.

The state machine stays in sync.

Stage processor worker

Firstly, why does the Stage Processor exist?

The API’s job ends at 202 Accepted. It validated the data, locked the stage to PENDING, and wrote the outbox event. The actual KYC work - calling third-party verification services, analyzing documents, matching faces - has not happened yet. Something has to do it.

That something cannot be the API. KYC checks are slow, unpredictable, and failure-prone. Doing them synchronously inside an HTTP request would mean the client waits minutes for a response, the API’s connection pool gets exhausted holding long-running requests open, and a single slow third-party service degrades the entire API for everyone else hitting it at the same time.

So the work is decoupled entirely. The API signals intent by writing to the outbox. The Outbox Relay picks that up and pushes a message to RabbitMQ. The Stage Processor is the dedicated consumer of that queue - the process whose only job is to take that message, run the real verification work, write the result back to Postgres, and drive the stage chain forward.

It also owns the sequential unlock logic. When Stage 1 finishes, Stage 2 must start. There is no cron watching for that. The Stage Processor does it - in the same transaction that marks Stage 1 COMPLETED, it inserts Stage 2’s outbox event. The pipeline advances the instant the previous stage commits. No polling, no delay, no external scheduler.

Without the Stage Processor, the system has no execution engine. The data sits in PENDING forever and nothing ever moves.

What it does and how it flows

The muscle of the engine.

Consumes jobs from RabbitMQ, runs the verification logic, drives the stage chain forward, and handles every failure path without losing data.

Intake - triage before anything else

The worker is configured with QoS prefetch of 1 (configurable) - RabbitMQ only hands it one message at a time per thread.

No memory floods from large backlogs.

The moment a message arrives, it's triaged immediately.

If the payload is missing event_id, session_id, or stage, the message is rejected with requeue: false and sent straight to the dead-letter queue.

Garbage is never retried.

Claim - idempotency and lease

The worker tries to claim the stage by updating its status to PROCESSING and writing an expires_at 120 (configurable) seconds out.

If Postgres returns 0 rows updated, another worker already claimed it or it's already completed - the worker ACKs the RabbitMQ message to clear it and moves on.

No duplicate work, no errors.

Execution - concurrent jobs, isolated traces

Jobs run concurrently inside a JoinSet.

Each job gets its own tracing span so logs from 50 concurrent tasks never bleed into each other.

A failure in Job A produces a clean, isolated trace.

Stage handoff - decentralized orchestration

When a stage completes, the worker checks inside the same transaction whether a successor stage is waiting.

If it is, the outbox event for the next stage is inserted before the commit.

Stage 2 is triggered the instant Stage 1 commits - no cron job, no external scheduler, no delay.

Session sweep - global state stays in sync

After every completion or failure, the worker checks all five stages for the session.

If all five are COMPLETED, the session is promoted to COMPLETED.

If one is permanently FAILED, the session becomes FAILED.

This happens in the same transaction as the stage update.

Failure path - retries then graveyard

If the KYC work fails, the stage is set to FAILED and retry_count is incremented.

If retries remain, the payload is published to the retry exchange - which holds it for 30 seconds before routing it back to the main queue - and the original message is ACKed.

If retries are exhausted, the message is sent to the terminal dead-letter queue and the stage stays permanently FAILED.

Graceful shutdown - no mid-job corruption

On SIGTERM, the worker stops pulling new messages immediately.

It then waits for all in-flight jobs in the JoinSet to finish before the process exits.

A deployment never kills a job halfway through.

End-to-end walkthrough

Phase 1 - Session start (POST /session/start)

The API key is checked in constant time.

If an idempotency key is present, Redis is checked first - a duplicate returns the cached response immediately.

Otherwise, one transaction opens: one session row (ACTIVE) and five stage rows (LOCKED) are inserted together.

Transaction commits, result is cached in Redis for 2 hours, 201 Created returned with the session UUID.

Phase 2 - Stage submission (POST /stages/{name})

Payload hits ValidatedJson - Serde validates structure, validator enforces field rules, libphonenumber validates phone format where applicable.

Any failure returns 422 with field-level detail.

Inside a transaction, the stage is updated from LOCKED to PENDING.

Zero rows affected means it was already submitted - 409 Conflict returned, nothing else happens.

Then the dependency fork: independent stages (1, 4, 5) get an outbox_events row inserted immediately.

Dependent stages (2, 3) check if their predecessor is COMPLETED - if yes, outbox event is inserted; if no, the insert is skipped and the stage waits.

Transaction commits, 202 Accepted returned.

Phase 3 - Outbox relay

Every 500ms the relay fetches up to 50 rows using SELECT FOR UPDATE SKIP LOCKED from its own isolated pool.

Multiple relay pods grab different rows simultaneously, never blocking each other.

Each row is published to RabbitMQ under a strict 50ms timeout.

If the connection hangs, the timeout fires, the row is skipped, and the lock is released - database health over message speed.

Successfully ACKed message IDs are collected, bulk-deleted from the outbox, and the transaction commits.

Phase 4 - Stage processor

Worker consumes with prefetch of 1.

Malformed payload is immediately rejected to the DLQ - no retries on garbage.

Valid payload opens a transaction, claims the stage (PROCESSING), and sets expires_at.

Success path: Stage is updated to COMPLETED with response_data.

If a successor stage is PENDING, its outbox event is inserted in the same transaction.

Session sweep runs - all five completed means the session becomes COMPLETED.

Transaction commits, RabbitMQ ACK sent.

Failure path: Stage is updated to FAILED, error saved, retry_count incremented.

If under the retry limit, the payload goes to the retry exchange (30s hold, then back to main queue) and the original message is ACKed.

If retries are exhausted, the message is rejected to the terminal dead-letter queue and the stage stays permanently FAILED.

Session sweep runs in both cases.

Phase 5 - Lease recovery

Every 60 seconds the worker queries for PROCESSING rows where expires_at < NOW(), in batches of 25 using SKIP LOCKED.

If lease_recovery_count is under the limit: stage reset to PENDING, count incremented, fresh outbox event inserted - the job re-enters the pipeline.

If at the limit: stage permanently set to FAILED, no outbox event inserted, session sweep runs.

A payload that crashes the worker every time is quarantined permanently.

Everything happens in one transaction - a database crash mid-recovery leaves the rows expired and untouched until the next tick.

Phase 6 - Status poll (GET /status)

API key checked.

Two indexed queries run - one for the session row, one for all five stage rows.

Results mapped into SessionStatusResponse and returned as 200 OK.

The core guarantee. The database is the source of truth.

RabbitMQ is purely a signal.

The transactional outbox, SKIP LOCKED concurrency, timed leases, and three-tier dead-letter topology together guarantee that every stage is eventually processed - regardless of network conditions, pod failures, or duplicate message delivery.

High-level architecture

Three workers. One clear job each

Before anything else - this system runs exactly three background workers, and none of them overlap in responsibility:

Outbox relay Bridges Postgres to RabbitMQ
Stage processor Consumes from RabbitMQ and drives verification logic
Lease recovery Cleans up crashed jobs, runs entirely out-of-band

That's the foundation.

Every data path in this system flows through one of those three, or the API itself.

The topology

Edge → API layer

The client hits the API over HTTP/REST.

The API talks to exactly two things: Redis to validate the idempotency key, and Postgres to write the session and seed the outbox_events table transactionally.

The API never touches RabbitMQ - that boundary is intentional.

Redis

Exclusive to the API layer. Workers never interact with it.

PostgreSQL

The central nervous system.

Four things connect to it: the API (writes sessions and outbox rows); the Outbox Relay Worker (reads and deletes outbox rows); the Stage Processor Worker (updates stage statuses, inserts next outbox row on stage unlock); the Lease Recovery Worker (scans for zombies, resets them, re-inserts outbox rows).

Outbox relay worker

Polls outbox_events from Postgres via its own isolated connection pool, then pushes those events into RabbitMQ over TCP.

It's the only component that writes to the broker.

RabbitMQ

Sits entirely between the workers.

Receives from the Outbox Relay, delivers to the Stage Processor.

Holds the Main, Retry, and Dead-Letter queues.

Stage processor worker

Consumes from RabbitMQ with a strict prefetch of 1.

Updates stage status in Postgres.

If a completed stage unlocks the next one, it writes the successor's outbox_event directly back into Postgres - feeding the relay again.

Lease recovery worker

Postgres only. No RabbitMQ.

Scans for timed-out or crashed jobs, resets their state, and inserts fresh outbox_events - which naturally flow back through the Outbox Relay as if nothing crashed.

API architecture

Foundation: one process, many concurrent requests

Before diving into the layers - the API is a single OS process running a Tokio async runtime.

There are no threads per request, no forking, no copying of state.

Tokio multiplexes thousands of concurrent connections across a small fixed thread pool.

Everything below is built on top of that single fact.

The HTTP engine - Hyper & Axum

Axum is not a heavy framework.

It's a thin routing layer on top of Hyper, which handles the raw TCP byte stream and HTTP header parsing.

Every route is a tower::Service - a typed async function from Request to Response.

Routing a request to a handler is a pointer lookup, nothing more.

No reflection, no magic, no overhead.

Shared memory - Arc<AppState>

The Postgres pool, Redis manager, AMQP connection, and config all live in one AppState struct, wrapped in an Arc.

Ten thousand concurrent requests don't get ten thousand copies - they get ten thousand pointers to the same struct, backed by a single atomic counter increment.

One allocation at boot, zero copies at runtime.

The interceptors - FromRequest

Validation never lives inside handlers.

ValidatedJson<T> implements FromRequest, so Axum runs it before the handler function is even scheduled.

If Serde parsing or field validation fails, the extractor returns a 400 or 422 directly and the handler never wakes up.

The edge is the enforcement point.

The middleware onion - Tower layers

Every request passes through a stack of tower::Layers before it reaches the router.

The outermost layer is TraceLayer - it creates a tracing span and stamps a trace_id onto the async context.

Every DB query, Redis call, and log line emitted deeper in the stack inherits that ID automatically.

No manual propagation needed.

The database multiplexer - PgPool

Postgres connections are opened once at boot, held in PgPool, and leased per request via pool.begin().await.

When the transaction commits or the variable drops out of scope, the connection returns to the pool immediately.

This is what lets 20 Postgres connections serve thousands of concurrent HTTP requests without contention.

Request contract

URL design - resource-oriented slugs

POST /v1/verify/{session_id}/stages/{stage_name}
  • Every segment is deliberate. /v1/ is explicit versioning - when a v2 flow ships with 10 stages, both versions run concurrently without breaking existing clients.
  • {session_id} in the path rather than the body lets the router and log infrastructure (CloudWatch) index by session ID without parsing JSON payloads.

Request contract - zero trust at the border

The principle: assume every incoming request is malformed or malicious until the type system proves otherwise. Business logic never sees bad data.

  • ValidatedJson<T> runs a two-pass check before the handler wakes up.

    The first pass is structural - does the JSON deserialize into the expected Rust struct?

    The second pass is semantic - is that phone number actually valid for Nigeria or the UK?

    Both passes must succeed.

    Failure at either returns a 400 or 422 immediately at the extractor layer.

  • x-api-key authentication uses the subtle crate for constant-time comparison.

    Standard string equality (==) short-circuits and returns faster when early characters match - an attacker can measure response times and progressively guess the key character by character.

    Constant-time comparison takes identical time regardless of how many characters match, eliminating that attack surface entirely.

  • Idempotency-Key is optional on session start.

    Once present, it gates duplicate submissions at the Redis layer before anything touches the database.

Response contract

Predictable by design

Every response from this API, success or failure, follows a fixed shape. Clients never have to guess what they'll get back.

Success (201/202)

{
  "session_id": "uuid",
  "status": "PENDING",
  "message": "Human readable hint"
}

Error (4xx/5xx)

{
  "error": "validation_failed",
  "message": "One or more fields failed validation",
  "trace_id": "016x_hex_string",
  "details": { "phone": "Invalid format" }
}

  • trace_id on every error. When a frontend engineer reports a 500, you don't ask for a screenshot - you ask for the trace_id.

    Paste it into your logs and you get the exact database queries, Redis checks, and RabbitMQ interactions for that specific request, down to the millisecond.

    No guessing, no reproduction steps needed.

  • 202 Accepted instead of 200 OK. Stage submissions return 202 because 200 implies the work is done.

    It isn't - the KYC processing happens asynchronously via RabbitMQ workers.

    202 is an explicit contract with the client: "We received it, validated it, and queued it. The engine is still running."

  • Error masking on 500s. The real error - relation "users" does not exist, a raw Postgres panic - is logged internally.

    The client only sees "An unexpected error occurred".

    Exposing internal error messages leaks your database schema, table names, and system internals to anyone probing the API.

Worker architecture

Foundation: workers are not threads

Before the layers - each worker is a single long-running Tokio async task, not a thread pool spinning up per job.

There's no thread-per-message model here.

Tokio's event loop cooperatively schedules everything, and the primitives below are what make that safe, concurrent, and survivable under real infrastructure pressure.

The async engine - Tokio & JoinSets

Each worker runs an infinite loop {} inside a Tokio task.

When the Stage Processor pulls multiple messages, it doesn't process them one-by-one, nor does it blindly tokio::spawn (which leaks if tasks hang).

Every message becomes a future pushed into a tokio::task::JoinSet.

The set fires them concurrently - the moment one task awaits network I/O, the CPU is yielded to another.

Parallelism without threads.

Graceful shutdown - CancellationToken

Kubernetes doesn't ask nicely - it sends SIGTERM and counts down.

Every loop and background task gets a clone of a tokio_util::sync::CancellationToken at boot.

When shutdown fires, cancel.cancel() triggers.

Every select! branch listening on that token stops pulling new messages from RabbitMQ.

The worker then drains its JoinSet, waiting for in-flight tasks to finish writing to Postgres before the process exits.

No mid-transaction kills.

The data backbone - SQLx transaction boundaries

No ORM.

Every worker operation that touches Postgres - claim a job, unlock a successor, sweep a session - opens one pool.begin().await and passes &mut tx through every call.

Either everything commits, or the connection drops and the entire block rolls back atomically.

There's no partial state possible.

Messaging - Lapin & explicit ACKs

A single lapin::Connection over TCP multiplexes lightweight virtual Channels.

Workers consume with manual acknowledgement - the Delivery object is held in memory until the SQLx transaction commits successfully.

Only then does delivery.basic_ack() fire.

If the payload is malformed, basic_reject(requeue: false) severs the message and routes it to the dead-letter queue.

Nothing is auto-acked, nothing silently disappears.

Time control - timeouts & intervals

Distributed systems hang before they crash.

Every external call is wrapped in tokio::time::timeout - RabbitMQ publishes get a hard 50ms ceiling; if the TCP socket silently drops packets, the future is aborted and an error is returned rather than a hung thread.

Polling workers use tokio::time::interval() with MissedTickBehavior::Delay - if a Postgres fetch runs long under load, the worker waits a full fresh interval before querying again.

It doesn't try to catch up.

That's the pressure release valve.

Thought process

One rule - keep business logic in one place

The entire structure is driven by a single architectural decision: crates/core owns everything, apps/ owns nothing.

The two binaries (api and worker) are deliberately kept dumb - they boot Tokio, wire infrastructure, and delegate.

All business rules, SQL, validation, and worker logic live in core.

This means you can test the full domain without spinning up an HTTP server.

You can scale the API and workers independently as separate OS processes without duplicating a single line of logic.

  • Root level

    The project/ folder is the Cargo workspace root.

    Cargo.toml pins all shared dependency versions once - individual crates just inherit them.

    docker-compose.yml and Makefile live here so every make and docker compose command resolves paths correctly from this single working directory.

  • apps/ - the two thin binaries

    These are the deployable processes.

    Neither contains business logic.

    apps/api/ - Boots Tokio, constructs Arc<AppState>, mounts the Axum router from core, and listens on the configured port.

    app.rs wires the Tower middleware stack; router.rs maps URL paths to handlers defined in core.

    apps/worker/ - Boots Tokio, constructs Arc<AppState>, and starts the three background workers registered in registry.rs.

    Nothing else.

  • crates/core/src/ - the brain

    Everything domain-critical lives here, organized by concern.

    • shared/

      Cross-cutting infrastructure.

      config.rs - typed AppConfig from env via envy.

      state.rs - Arc<AppState> with pool, Redis, AMQP, config.

      error.rs - central AppError; all errors convert here before HTTP responses.

      extract.rs - ValidatedJson extractor.

      redis.rs - Redis wiring and idempotency helpers.

      tracing.rs - subscriber init; JSON in prod, pretty locally.

    • verification/

      Vertical slice.

      http.rs - Axum handlers and DTOs (edge only).

      service.rs - orchestration; db + messaging; business rules.

      db.rs - raw SQLx on &PgPool or &mut Transaction.

      types.rs - VerificationSession, StageStatus, etc.

      helpers.rs - pure in-slice functions.

      openapi.rs - Utoipa schemas for compile-time OpenAPI.

    • messaging/

      RabbitMQ isolated; slices call publish_stage_event(...) etc. - no scattered routing keys.

      topology.rs - exchanges and queues in one place.

      publisher.rs - typed publish for service and workers.

    • workers/

      outbox_relay.rs - poll outbox_events, publish to RabbitMQ.

      stage_processor.rs - consume, drive stage transitions in Postgres.

      stage_lease_recovery.rs - zombie jobs, out-of-band reset.

      jitter.rs - shared retry jitter.

  • crates/core/tests/

    unit/ - Pure logic: config, errors, extractor, types - no I/O.

    integration/ - E2E against real Postgres, Redis, and RabbitMQ; one file per slice.

    support/ - harnesses and fixtures shared by integration tests.

  • migrations/

    Ordered, append-only.

    Applied once by sqlx migrate run.

    Sequence: sessions → stages → outbox → lease → recovery index.

    Never edited after apply.

  • devops/

    One Dockerfile per process (api, worker, migrations, test).

    Each image purpose-built and minimal.

  • scripts/

    verification-stress.mjs - parallel sessions, CSV reports; tooling, not app code.

Data architecture

verification_sessions table

Tracks the top-level lifecycle of a verification attempt.

UUID primary key prevents enumeration.

status is the terminal state machine: ACTIVE, COMPLETED, or FAILED.

completed_at and failed_at are separate nullable columns - not a single ended_at - because how it ended matters.

updated_at is trigger-managed, always accurate.

idx_sessions_status covers any sweep filtering by status.

verification_stages table

One row per verification step (1-5) within a session.

ON DELETE CASCADE means session deletion cleans up all stages automatically.

stage is a SMALLINT - smaller, faster index lookups.

Status follows LOCKED → PENDING → PROCESSING → COMPLETED | FAILED.

input_data is written once by the API, never touched again.

response_data is written only by the worker on completion.

Two writers, two columns - the audit trail is structurally immutable.

retry_count and error track failure history; error is cleared on a successful retry.

expires_at turns a job claim into a timed lease.

A crashed worker pod can't leave a stage stuck in PROCESSING forever - the timestamp expires and the recovery worker reclaims it.

lease_recovery_count prevents infinite recovery loops on top of that - a toxic payload that crashes the worker every time would be reset to PENDING forever without it.

Once it hits the configured limit, the stage is permanently FAILED and abandoned.

uq_session_stage makes duplicate stage rows physically impossible.

chk_valid_stage rejects out-of-range values at the DB level regardless of what the application sends.

Three indexes: idx_stages_polling - partial on PENDING/PROCESSING only, so worker polling never scans completed rows.

idx_stages_session - covers the sweep query reading all stages for a given session.

idx_stages_processing_expires_at - partial on PROCESSING rows with a non-null expires_at, so expired lease lookups by the recovery worker are always an index scan, never a table scan.

outbox_events table

The transactional outbox bridge between Postgres and RabbitMQ.

Stores exchange, routing_key, and payload (JSONB including the worker's idempotency event_id).

Rows are hard-deleted after the relay gets an AMQP ack - never updated, not an audit log, designed to stay near-empty.

idx_outbox_polling on created_at ASC keeps oldest-first polling instant even during traffic spikes.

UUIDv4 over BIGSERIAL - why

Sequential integers are fine for simple CRUD apps. For a distributed KYC engine handling sensitive identity data, they're a security flaw.

  • IDOR immunity.

    With id = 100, a malicious script trivially guesses id = 101 and scrapes other users' verification states.

    UUIDv4 carries 122 bits of entropy - guessing one is not computationally feasible.

    The ID itself is a security layer.

  • Bearer token model.

    The verification tables store no user_id - the session UUID is the credential.

    The client holds it, presents it, and the engine trusts it.

    For a URL parameter to function as a bearer token, it must be unguessable by definition.

  • No database round-trips for ID generation.

    Sequential IDs require a round-trip to claim the next value from a locked database sequence - a concurrency bottleneck at high throughput.

    UUIDs are generated entirely in memory: Uuid::new_v4() in Rust, inline, with no DB interaction.

    Multi-stage transactions don't stall waiting for an ID to come back.

  • Distributed uniqueness.

    The transactional outbox relays events across workers and pods.

    Sequential IDs from two database shards would collide - both would produce id = 5.

    A UUIDv4 generated in Pod A will never collide with one generated in Pod B, ever.

The trade-off. 16 bytes versus 8 bytes for a BIGINT, and B-tree indexes on random UUIDs fragment slightly more.

That cost is accepted - a few extra megabytes of storage in exchange for IDOR immunity, stateless ID generation, and collision-free distributed operation.

Message queue architecture

Foundation: three tiers, one rule - no message is ever lost

Every message in this system has exactly three possible fates: it succeeds, it retries, or it lands in a graveyard for human inspection.

There is no fourth option where it silently disappears.

The publishers

  • Messages are published with delivery mode 2 - persistent.

    RabbitMQ writes them to disk before acknowledging.

    A broker crash or restart does not lose in-flight messages.

  • Publishing uses a double-await pattern.

    The first await sends the bytes over TCP.

    The second await blocks until RabbitMQ sends back an explicit confirmation that the message has been safely received.

    No fire-and-forget, no silent data loss.

  • When a failed message is moved to the retry queue, the existing x-death headers are preserved and carried forward.

    These headers track how many times a message has failed across the entire system - without them, the worker has no way to enforce a max-retry limit.

  • The payload itself is kept deliberately minimal - just UUIDs and a stage number.

    No PII, no full database rows.

    If the worker needs more context, it queries Postgres using those IDs.

    This keeps queues lightweight and eliminates the class of bugs where a worker acts on stale data embedded in a message.

The three-tier topology

The topology is declared idempotently on every boot.

If the exchanges and queues already exist, RabbitMQ ignores the declarations.

If they don't, it creates them.

The infrastructure heals itself automatically - no manual setup required after a fresh environment.

All exchanges use Direct routing - one message, one consumer.

No fanout, no broadcast.

Tier 1 - Main queue

Where active jobs live.

Workers consume from here.

When a worker hits max retries and calls basic_reject(requeue=false), RabbitMQ doesn't drop the message - the x-dead-letter-exchange configuration automatically routes it to the terminal exchange instead.

Tier 2 - Wait queue

The sleep queue.

No workers consume from it.

Failed messages that haven't yet hit the retry limit land here and sit for exactly 30 seconds via x-message-ttl.

When the TTL expires, RabbitMQ itself routes the message back to the main queue using the configured dead-letter exchange and routing key.

Fully infrastructure-driven retry backoff - no sleep timers, no scheduled tasks, no Rust code involved.

Tier 3 - Dead queue

The graveyard.

No TTL, no workers, no automatic routing.

Poison messages - ones that crash the worker every time, or ones rejected after hitting the retry ceiling - land here permanently.

A human engineer can inspect them in the RabbitMQ management UI, diagnose the root cause, fix the bug, and manually shovel them back to the main queue when ready.

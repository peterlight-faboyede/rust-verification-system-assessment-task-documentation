DevOps

Build strategy, deployment safety, migrations, and test image workflow.

API & worker images

Both use a multi-stage cargo-chef build. Chef separates dependency compilation from application compilation into distinct Docker layers. The first build is slow - all 200+ dependencies compile from scratch. Every subsequent build skips that layer entirely and only recompiles changed application code. A code change that previously took 10 minutes now takes roughly 10 seconds.

The build base is pinned to bookworm to match the runtime image (distroless/cc-debian12). Mismatched glibc versions between build and runtime cause immediate crashes - a binary compiled against a newer glibc simply won't run in the older distroless container. The versions are locked together deliberately.

SQLX_OFFLINE=true means SQL queries are verified against committed JSON metadata instead of a live database. CI pipelines don't need a running Postgres instance to compile - no flaky pipeline failures from missing database connections.

The final images are distroless - no shell, no package manager, no system utilities, roughly 50MB. If a zero-day is exploited, there are no tools available inside the container to do anything with it.

API and worker are separate images built from the same source. They scale independently in Kubernetes - more worker pods for processing backlogs, without spinning up unnecessary API servers alongside them.

Migrations image

Migrations run in a dedicated ephemeral container, not inside the API startup code. Ten API pods booting simultaneously and all trying to migrate would deadlock each other. The migration container runs first, exits with code 0, and only then does the API rollout begin.

sqlx-cli is compiled against musl and statically linked, then dropped into a plain Alpine image. No dynamic library dependencies. ca-certificates is installed explicitly - without it, the binary can't verify SSL certificates against managed cloud databases like RDS or Supabase. Only the Postgres feature is compiled in - no MySQL or SQLite drivers bloating the binary. Final image is around 30MB.

Test image

cargo test --no-run runs during the image build, forcing all test binaries to compile and bake into the Docker layer. When the tester container runs in CI, tests execute immediately - no compilation wait at test time. The final stage retains the full Rust toolchain so the container can be used interactively during development.

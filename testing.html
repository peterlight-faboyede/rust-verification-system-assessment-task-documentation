Foundation: three testing worlds, three purposes

The test suite is divided into three completely separate environments. Each exists for a different reason and tests a different level of the system.

Unit tests run entirely in memory. No database, no Redis, no RabbitMQ. They test pure logic - config parsing, error mapping, validation boundaries, type serialization. If a unit test needs infrastructure to run, it's testing the wrong thing.

Integration tests run against real infrastructure. Every test gets its own ephemeral Postgres database, spun up and torn down automatically. They test that the actual SQL, the actual locks, the actual business rules, and the actual HTTP layer all behave correctly when wired together. Mocks are deliberately avoided here - the point is to prove the system works against real Postgres transactions and real Redis state.

Stress tests run against the full live cluster - API, worker, Postgres, Redis, RabbitMQ all running simultaneously. They don't test correctness; they test behavior under load. The goal is to find where the system bends before it breaks in production.

Unit tests - pure logic, zero infrastructure

  • Config validation - proves that misconfigured environment variables (min delay higher than max, malformed CORS URLs) crash the app at boot, not silently in production.
  • Error mapping - proves every internal error variant maps to the correct HTTP status code and JSON shape. A database RowNotFound never leaks raw SQL text to the client.
  • ValidatedJson extractor - proves the exact boundary between a 400 (broken JSON structure) and a 422 (valid JSON, failed business rules).
  • Redis key formatting - proves idempotency keys are constructed correctly and HTTP header extraction works regardless of casing.
  • Stage helpers - proves the predecessor/successor chain is correct. Stage 3 cannot unlock Stage 5. The dependency graph has no gaps or shortcuts.
  • Phone validation - proves libphonenumber catches invalid international numbers before they reach the database.
  • Payload struct validation - proves empty base64 strings, impossible dates of birth, and missing required fields all fail validation. Enums serialize and deserialize in strict snake_case only.

Integration tests - real infrastructure, real behavior

  • Database layer - proves the raw SQL works before anything else. FOR UPDATE SKIP LOCKED is tested directly. The session sweep logic is proven: one failed stage correctly flips the parent session to FAILED.
  • Outbox relay - a scripted publisher simulates TCP hangs, publish timeouts, and corrupt payloads. Proves that when RabbitMQ rejects a publish, the Postgres transaction rolls back and the outbox row survives intact. A 10,000-row batch test proves the indexes keep queries under 5 seconds.
  • HTTP layer - the full Axum router is wired with a live AppState. Tests bombard it with missing API keys (401), malformed JSON (400), and business validation failures (422). The idempotency key is tested against live Redis - double-submitting session start must not create duplicate database rows.
  • Service logic - proves the sequential dependency chain. Stage 1 submission hits the outbox immediately. Stage 2 submission does not hit the outbox because Stage 1 isn't complete yet. The dependency tree is enforced at the database level.
  • Lease recovery - Tokio's time control is used to simulate the 60-second recovery loop without waiting 60 seconds in CI. A custom tracing layer intercepts internal spans to prove the worker sleeps correctly, resets its ticker on finding work, and doesn't thrash the database when idle. The poison pill defense is proven: three crashes result in permanent FAILED, no further recovery attempts.
  • Stage processor - payload parsing is tested without needing a live RabbitMQ connection. Worker idempotency is proven: claiming an already-completed stage is a safe no-op. The sequential unlock is proven: completing Stage 1 inside a transaction automatically inserts the Stage 2 outbox event.

Stress tests - full cluster under load

Every verification session is launched at the exact same millisecond using Promise.allSettled - a deliberate thundering herd. This forces Postgres row locks, outbox micro-batching, and RabbitMQ routing to prove themselves under maximum contention simultaneously.

The script mounts the Docker socket and acts as its own infrastructure monitor - querying real-time CPU, RAM, and network I/O for every container every 3 seconds. Network stats are delta-computed so the numbers reflect bandwidth during the test window specifically, not since the container booted. A slow response can be traced directly to a container hitting 100% CPU.

Results are written to three CSV files:

  • Session-level outcomes - pass/fail result for every verification session in the run.
  • Per-stage round-trip and wall-clock timing - granular latency for each of the five stages across all sessions.
  • Hardware telemetry - CPU, RAM, and network I/O mapped against elapsed test seconds for every container.

Internal status codes are translated to plain English before writing - LOCKED becomes "Not started yet" - so results are readable without understanding the state machine.

A hard deadline backed by an AbortController guarantees the script exits cleanly even during a catastrophic cluster failure. Whatever data was collected up to that point is written to CSV. Test data is never lost.

Node.js was chosen over Rust for the stress script - native fetch, async event loop, and JSON/CSV handling without compilation overhead. It's the right tool for firing concurrent HTTP traffic and wrangling reports.

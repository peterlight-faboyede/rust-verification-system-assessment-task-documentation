Runtime & HTTP

  • Tokio - async runtime, the concurrency foundation. async-std was skipped due to weaker ecosystem support.
  • Axum - HTTP routing with strong type safety, sits naturally on Tower. actix-web was ruled out for being more actor-oriented and less aligned here.
  • Tower / tower-http / tower_governor - middleware as composable layers: CORS, tracing, request IDs, and rate limiting all live here, outside business logic where they belong.

Data & messaging

  • SQLx - async PostgreSQL with compile-time query validation. Diesel's DSL felt too restrictive for the fine-tuned SQL this system needs.
  • Lapin - AMQP client for RabbitMQ. Chosen for fine-grained control; higher-level wrappers hide too much.
  • Redis - idempotency checks and distributed locks. Fast coordination layer between workers and the API.

Serialization & validation

  • Serde + serde_json - standard, mature, no real alternative in Rust.
  • Validator - declarative rules on DTOs, enforced before any core logic runs.
  • phonenumber - international phone validation. Regex simply isn't reliable for global numbering rules.
  • UUID + Chrono - non-sequential IDs and consistent UTC timestamps throughout.

Observability & config

  • Tracing + tracing-subscriber - structured spans that survive async boundaries, formatted as JSON for observability pipelines.
  • Envy + dotenvy - typed config from environment variables, fails fast on missing values. dotenvy replaces the unmaintained dotenv.
  • Secrecy - wraps sensitive values to prevent accidental exposure in logs or memory.

Security & reliability

  • thiserror + anyhow - structured errors for domain logic, flexible errors for worker flows.
  • subtle - constant-time comparisons to prevent timing attacks on sensitive checks.
  • rand - secure token generation and retry jitter.
  • futures-util - stream and async utilities the standard library doesn't cover yet.

API docs

  • Utoipa + utoipa-swagger-ui - OpenAPI spec generated at compile time, served as an interactive Swagger UI. Manual Swagger files drift; this keeps docs and code in sync automatically.

Tokio as the runtime - why

Async-first ecosystem alignment. Every library in this stack - sqlx, lapin, redis, axum - is built on Tokio. They share the same async primitives, the same executor, and the same task model. No bridging, no compatibility shims, no runtime conflicts.

Non-blocking async I/O. Tokio multiplexes thousands of concurrent connections - database queries, RabbitMQ publishes, HTTP requests - across a small fixed thread pool. No thread-per-request model, no OS thread overhead. The workers and API can handle massive concurrency without proportional memory cost.

select! and CancellationToken for shutdown. Graceful shutdown across every worker loop is built around tokio::select! and CancellationToken. These are Tokio primitives - clean, composable, and cooperative with the async runtime rather than fighting it.

JoinSet for bounded concurrency. The Stage Processor runs multiple jobs concurrently inside a JoinSet. This gives structured concurrency - tasks are tracked, drained on shutdown, and never leak - which is not possible with raw spawn calls.

Time control in tests. tokio::time::pause() and tokio::time::advance() let the Lease Recovery tests simulate a 60-second polling loop in milliseconds. No real waiting, no slow CI. This is only possible because Tokio owns the time abstraction inside the async runtime.

Axum + Tower - why

Native Tokio integration. Axum is built by the Tokio team and runs directly on Tokio’s runtime. No adapter layers, no compatibility shims - it shares the same async executor as every other dependency in the stack.

Every route is a tower::Service. This means middleware - rate limiting, tracing, CORS, request IDs - is composed as standard Tower layers, completely outside business logic. The same middleware primitives work across the entire HTTP stack without framework-specific glue.

Compile-time extractor safety. FromRequest and FromRequestParts enforce validation and authentication at the type level. A handler that requires ValidatedJson<T> or ValidatedClient literally cannot compile without those extractors being correctly wired. Errors are caught before the binary runs, not at request time.

Zero framework overhead. Axum adds no reflection, no runtime magic, no macro-heavy annotation scanning. Routing is a pointer lookup. The abstraction cost is zero - raw Hyper performance with structured routing on top.

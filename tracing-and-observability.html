Boot setup

Both the API and worker call one initialization function at startup. Log level is controlled by RUST_LOG - defaults to info in production to keep log volume and cost low. Injecting RUST_LOG=info,app_core=debug at runtime enables deep traces for a specific crate without redeployment.

Log format switches on environment. Local gets human-readable output with file names and line numbers - click a log line, jump to the code. Production gets compact JSON - lower CPU overhead, easy for log aggregators to parse.

Spans over log lines

Everything is structured around spans, not individual log statements. A span covers a duration and carries context - session_id, stage, outbox_id. Every log event emitted inside a span automatically inherits that context. You don't pass variables through every function call; the span carries them.

trace_id on every API error

Tower's trace middleware stamps a unique 16-character hex ID on every HTTP request. When the API returns a 4xx or 5xx, that ID is injected directly into the JSON error response. A customer pastes that ID into the log search and the exact database query, validation failure, and stack trace for their specific request appears instantly. No reproduction steps needed.

Worker spans built manually

Workers have no incoming HTTP request, so spans are constructed explicitly. The outbox relay extracts session_id and stage from the payload before publishing and attaches them to the span - a failed publish log reads "failed to publish Stage 2 for Session XYZ", not just "publish failed."

The stage processor uses .instrument(span) on each async task in the JoinSet. Standard span entry would bleed log context across concurrently running jobs sharing the same thread. .instrument() binds the span to the specific future, not the thread - Job A's logs never mix with Job B's.

Surgical field recording

Instead of logging generic completion messages, specific counts are recorded directly onto the span - recovered jobs, permanently failed jobs. Log aggregators can graph these fields in real time without custom parsing rules.

Debug gate

Expensive log operations - like printing a list of all published outbox IDs - are gated behind tracing::enabled!(Level::DEBUG). At info level, that code path is never evaluated. Debug detail is available when needed without paying the cost in production.
